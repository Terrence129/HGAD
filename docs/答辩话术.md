**你的方法怎么讲**
第一步，数据处理。SMD 每台机器有训练集、测试集和 label。你只用训练集 fit `StandardScaler`，再 transform 测试集，这样避免数据泄露。然后用 `window=10, stride=1` 把时间序列切成窗口；如果一个窗口里任意时间点有异常，该窗口就标记为异常。

第二步，模型输入。对 LSTM 来说，输入是 `(window, features)`，即时间序列窗口。对 GCN/HGNN 来说，你把每个 metric 当成一个 node，节点特征是这个指标在 10 个时间步内的值，所以输入变成 `(38 nodes, 10 time values)`。

第三步，图结构。GCN 用 pairwise correlation adjacency，只能表达两两关系。HGNN 用 incidence matrix `H`，一条 hyperedge 可以连接多个指标，因此更适合表达“多个监控指标一起异常”的 cloud fault 场景。你的 batch 实验里主要用了 global hypergraph：从训练集相关性构建，比较稳定；代码也支持 global + local，但最终大规模实验为了稳定性使用 global。

第四步，异常分数。模型都是 autoencoder 思路：在正常训练数据上学习重构正常行为。测试时，如果 reconstruction error 高，就认为异常。HGNN 还加入了 temporal error 和 embedding deviation，最终 score 是 weighted combination。threshold 不是固定手写，而是从训练分数的 percentile 中搜索，例如 96、97、98、99、99.5。

**老师可能问：为什么用 hypergraph？**
回答：

普通 graph 只能表达 pairwise relation，比如 metric A 和 metric B 相关。但云系统故障经常不是两个指标单独变化，而是一组指标共同变化，比如 CPU、memory、latency、I/O 同时异常。Hypergraph 的一条 hyperedge 可以连接多个 metrics，所以更适合表达 group-level dependency。我的模型用 hypergraph convolution 在这些 metric groups 上传播信息，再通过 reconstruction error 检测异常。

**老师可能问：为什么 HGNN 的 F1 没有 LSTM 高？**
回答：

这是一个 precision-recall trade-off。HGNN-AE 的 recall 是最高的，说明它能捕捉更多真实异常；但 precision 较低，说明 false positives 更多，所以 F1 被拉低。原因可能是 hypergraph 结构更敏感，对复杂相关变化更容易报警。我的结论不是 HGNN 已经完全优于 LSTM，而是它在高召回和诊断解释方面有价值，后续需要更好的 threshold calibration 和 dynamic graph tuning。

**老师可能问：你的 novelty 在哪里？**
回答：

我的 novelty 主要在工程实现和组合设计上：第一，把 SMD 多变量时间序列转成 metric-node reconstruction task；第二，用 correlation-based hypergraph 表达 higher-order metric dependency；第三，在 HGNN autoencoder 中加入 adaptive edge gates，让模型学习不同 hyperedges 的重要性；第四，不只输出 anomaly detection，还根据 metric-level reconstruction error 做 root-cause style ranking。

**老师可能问：为什么 window=10？**
回答：

Window size 10 是一个折中：太短会丢失 temporal pattern，太长会增加训练成本并让异常定位变粗。SMD 是高频监控数据，window=10 可以保留短期趋势，同时保持模型轻量，适合在 28 台机器上批量实验。

**老师可能问：有没有数据泄露？**
回答：

没有。Scaler 只在 training data 上 fit，然后用于 test data。Graph/hypergraph 的 global structure 也是从 training data correlation 构建的。测试标签只用于 evaluation，不参与模型训练。

**老师可能问：root cause detection 是真正的根因吗？**
回答要稳一点：

Strictly speaking, it is not verified causal root cause. It is a root-cause style ranking based on metric-level reconstruction and temporal errors. It tells us which metrics contributed most to the anomaly score, so it improves interpretability, but true causal validation would require labelled root-cause data or operator logs. This is one of my future work directions.

**老师可能问：如果部署到真实云系统怎么办？**
回答：

I would not deploy HGNN-AE as the only real-time detector at first. A practical deployment would use a fast detector for first-stage alerting, then use HGNN-AE as a second-stage diagnostic model for severe or ambiguous incidents. This balances efficiency, recall and interpretability.
